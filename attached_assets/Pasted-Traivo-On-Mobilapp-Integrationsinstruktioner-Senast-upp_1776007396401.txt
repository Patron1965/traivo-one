1. API-versionering — KRITISK ÄNDRING
Vad som ändrats
Alla REST-API-endpoints har nu ett versionsprefix: /api/v1/.

Vad Traivo On måste göra
Ändra alla API-anrop från /api/... till /api/v1/...
Exempel: /api/mobile/login → /api/v1/mobile/login
Exempel: /api/mobile/my-orders → /api/v1/mobile/orders/:id
Alla endpoints under /api/mobile/, /api/portal/, /api/notifications/ berörs
Bakåtkompatibilitet
Gamla /api/...-anrop fungerar fortfarande, men returnerar deprecation-headers:
Deprecation: true
Sunset: 2027-06-01
Link: </api/v1/...>; rel="successor-version"
Deadline: Unversioned endpoints fasas ut 2027-06-01
Versionsdiscovery-endpoint
GET /api/version
→ { "current": "v1", "supported": ["v1"], "deprecatedUnversioned": true, "sunset": "2027-06-01" }

Appen kan använda detta vid start för att verifiera kompatibilitet.

2. WebSocket Eventkatalog — Typat protokoll (v1)
Vad som ändrats
Alla WebSocket-events har nu formella Zod-validerade schemas med versionsfält (v: 1). Definitionen finns i shared/ws-events.ts och kan användas som kontrakt.

Anslutningsflöde
Steg 1 — Token Exchange:

POST /api/v1/notifications/token
Body: { "resourceId": "<resource-uuid>" }
→ { "token": "<ws-token>" }

Steg 2 — WebSocket-anslutning:

wss://<host>/ws/notifications?token=<ws-token>

Steg 3 — Heartbeat:

Skicka { "type": "ping" } var 30:e sekund
Servern svarar med { "type": "pong", "timestamp": "..." }
Steg 4 — Reconnect:

Vid disconnect: exponentiell backoff (10s → 20s → 40s, max 5 min)
Begär ny token för varje reconnect-försök
Server → App events (inkommande)
Alla notifieringsevents har denna basstruktur:

{
  "type": "<event-type>",
  "id": "uuid",
  "title": "Rubrik",
  "message": "Beskrivning",
  "orderId": "uuid (optional)",
  "resourceId": "uuid (optional)",
  "timestamp": "ISO-8601",
  "v": 1,
  "data": { ... }
}

Event-typ	Syfte	data-fält
connected	Handshake OK	message, timestamp
pong	Heartbeat-svar	timestamp
job_assigned	Nytt uppdrag tilldelat	scheduledDate, scheduledStartTime, objectName, objectAddress, priority
job_updated	Uppdrag ändrat	scheduledDate, scheduledStartTime, status
job_cancelled	Uppdrag borttaget	—
schedule_changed	Datum/tid ändrad	oldDate, newDate, scheduledStartTime
priority_changed	Prioritet ändrad	oldPriority, newPriority
position_update	GPS från annan resurs	latitude, longitude, speed, heading, status, workOrderId
route_update	Ruttändringar	—
order:updated	Generisk orderstatus	status, executionStatus, source
anomaly_alert	Systemvarning	—
route_optimized	Ruttoptimering klar	jobId
optimization_complete	Optimering slutförd (ny)	jobId
optimization_failed	Optimering misslyckades (ny)	jobId
notification	Generisk notis	valfria fält
App → Server events (utgående via WebSocket)
Event-typ	Syfte	Fält
ping	Heartbeat	—
position_update	GPS-telemetri	latitude, longitude, speed, heading, accuracy, status (traveling/on_site/idle), workOrderId
Rekommenderad hantering i appen
connected — Uppdatera UI till "online"-status
job_assigned — Visa push-notis + lägg till i uppdragslistan
job_updated / schedule_changed — Uppdatera lokal orderdata + notifiera användaren
job_cancelled — Ta bort från lokal lista + visa notis
optimization_complete — Ladda om uppdragslistan (schemat kan ha ändrats)
optimization_failed — Logga internt, ingen användaråtgärd krävs
order:updated — Synka lokal order-status

3.3 Exekveringsstatus-flöde
Statusövergångar via PATCH /mobile/orders/:id/status:

planned → dispatched → en_route → in_progress → completed
                                              → impossible
                                              → deferred

Status (skicka)	Effekt på servern
dispatched	executionStatus=dispatched, triggar ETA-notis till kund
en_route	executionStatus=on_way, sätter onWayAt, triggar ETA-notis
paborjad / in_progress	executionStatus=on_site, sätter onSiteAt
utford / completed	orderStatus=utford, executionStatus=completed, beräknar actualDuration
impossible	orderStatus=avbruten, kräver impossibleReason
ej_utford / deferred	Återställer till skapad, bifogar anteckning
3.4 Synkronisering (offline-first)

  "features": {

    "etaNotifications": true,

  },

Appen ska:

Dölja/visa funktioner baserat på features-flaggor

Visa "rekommenderad uppdatering" om under versions.recommended
Visa underhållsmeddelande om maintenance.enabled är true
5. Autentisering — tokenformat
Alla autentiserade endpoints kräver Authorization: Bearer <token> header
Token fås via POST /api/v1/mobile/login
Tokens är giltiga i 24 timmar
Vid 401-svar: logga ut användaren och visa inloggningsskärmen
Rate-limit: max 10 loginförsök per IP per 15 minuter (429-svar)
6. ETA-notifieringar till kunder
När appen sätter status till dispatched eller en_route triggar servern automatiskt ett ETA-SMS till kunden (om aktiverat för tenanten).

Appen behöver:

Skicka GPS-position regelbundet under körning (POST /api/v1/mobile/position)
Visa customerNotified: true i ordersvaret när SMS har skickats
Inget manuellt SMS-utskick behövs
7. Felhantering
Alla error-svar följer formatet:

{ "error": "Beskrivning av felet" }

Valideringsfel inkluderar detaljer:

{
  "error": "Valideringsfel: ...",
  "details": [{ "field": "namn", "message": "Fält krävs" }]
}

HTTP-statuskoder att hantera:

Kod	Betydelse	Åtgärd
200	OK	Bearbeta svaret
400	Valideringsfel	Visa felmeddelande
401	Ej autentiserad	Logga ut → inloggningsskärm
403	Ej behörig	Visa "Åtkomst nekad"
404	Ej hittad	Visa "Resursen finns inte"
429	Rate limit	Visa "Försök igen om X minuter"
500	Serverfel	Visa "Ett fel uppstod, försök igen"
